A filter sees an error coming out of a handler and turns it into a
`Response`. The success path passes through untouched. Two seams in
NestRS cover this — the `ResponseError` impl on a feature's error type
(the default), and `#[use_filters(...)]` for route-specific behavior.

```rust
#[async_trait]
pub trait Filter: Send + Sync + 'static {
    async fn filter(&self, req: &RequestSnapshot, error: poem::Error) -> Response;
}
```

## The default: framework error types, already mapped

The plumbing failures every service runs into — database errors,
input validation, authentication, OAuth wire codes — ship as framework
types with their HTTP mapping already wired:

| Error | Source | Maps to |
|-------|--------|---------|
| `ServiceError::Db(DbErr)` | `nest_rs_seaorm` | `500` (body: `"database error"`) |
| `ServiceError::Validation(ValidationErrors)` | `nest_rs_seaorm` | `422` |
| `AuthError::*` | `nest_rs_authn` | `401` + `WWW-Authenticate: Bearer` |
| `CredentialError` | `nest_rs_authn` | `401` (opaque `"invalid credentials"`) |
| `TokenError::*` | `nest_rs_authn::oauth` | RFC 6749 wire codes (`400` / `401` / `500`) |

A service returns the framework type directly; `?` flows it to the
boundary; the impl decides status and body. No per-feature `error.rs`
for these — the framework owns them once, every feature reuses them.

```rust
use nest_rs_seaorm::ServiceError;

impl UsersService {
    pub async fn create(&self, input: CreateUserInput) -> Result<User, ServiceError> {
        input.validate()?;                              // → ServiceError::Validation
        let row = active_for_new_user(input).insert(&Repo::<Users>::conn()?).await?;  // → ServiceError::Db
        Ok(User::from(&row))
    }
}

#[post("/")]
async fn create(&self, Valid(Json(input)): Valid<Json<CreateUserInput>>)
    -> Result<Json<User>, ServiceError>
{
    Ok(Json(self.svc.create(input).await?))
}
```

<Aside>
The data-layer HTTP mapping lives behind the `http` feature of
`nest-rs-seaorm`, so a worker or MCP-only app importing the crate
without that feature never links poem.
</Aside>

### When a feature defines its own error type

Write a per-feature error **only** when the failure is genuinely
domain-specific — a wire contract a consumer reads, a security-critical
opaque variant. In that case the type lives in `<feature>/error.rs`
and its `ResponseError` impl sits next to it (the features crate
already links poem because it ships HTTP controllers). Most features
never need this — they compose the framework's types.

## Route-bound filters via `#[use_filters(...)]`

Sometimes the mapping depends on the *route*, not the error. Maybe
`/v1/widgets/boom` should return an `I'm a teapot` while every other
500 stays a 500. That's what `#[use_filters(...)]` is for.

```rust
use nest_rs_core::injectable;
use nest_rs_http::{async_trait, Filter, RequestSnapshot};
use poem::http::StatusCode;
use poem::{Error, Response};

#[injectable]
#[derive(Default)]
pub struct TeapotFilter;

#[async_trait]
impl Filter for TeapotFilter {
    async fn filter(&self, _req: &RequestSnapshot, _error: Error) -> Response {
        Response::builder()
            .status(StatusCode::IM_A_TEAPOT)
            .body("filtered")
    }
}
```

Bind it per route or per controller:

```rust
#[controller(path = "/widgets")]
pub struct WidgetController;

#[routes]
impl WidgetController {
    #[get("/boom")]
    #[use_filters(TeapotFilter)]               // route scope
    async fn boom(&self) -> poem::Result<&'static str> {
        Err(Error::from_status(StatusCode::INTERNAL_SERVER_ERROR))
    }
}

#[controller(path = "/gadgets")]
#[use_filters(TeapotFilter)]                   // controller scope
pub struct GadgetController;
```

A filter sees a **`RequestSnapshot`** (method + URI + headers), not the
live `Request` — the inner endpoint has already consumed it. Snapshot
the routing-relevant bits up front, pass them to the renderer.

Filters are `#[injectable]` providers, listed in `providers = [...]` and
resolved from the container — same shape as guards and interceptors.

## Where filters sit in the chain

Per route, inner → outer:

```
shaper → interceptors → guards → filters → meta
```

A filter wraps *outside* guards and interceptors, so it sees errors
from any of them — a `403` from a guard, a `500` from a handler, an
error from an interceptor's `next.run(req)`. The successful path passes
through unchanged.

## Filters vs `ResponseError`

| Layer | Where it lives | When to use it |
|-------|----------------|-----------------|
| `ResponseError` on an error type | Framework crate that owns the error (`nest-rs-seaorm`, `nest-rs-authn`), or `<feature>/error.rs` for a domain-specific variant | Default. Every handler returning that error gets the same mapping |
| `#[use_filters(F)]` per route | A provider | Route-shaped behavior — a debug endpoint, a special status, a custom body envelope |

Reach for `ResponseError` first. A filter earns its keep only when the
status or shape depends on the route — not on the error itself.

## A small filter-only case: validation envelopes

The framework's pipe error renderer already returns:

```json
{ "statusCode": 400, "error": "Bad Request", "message": "...", "details": { /* ... */ } }
```

If your API uses a different envelope (`{ ok: false, error: { ... } }`),
write one filter that intercepts the error path and re-renders, bind it
globally. The error itself stays pure.
