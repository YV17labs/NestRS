A guard decides *whether* a handler runs. It sees the request before the
handler does and either lets it through (`Ok(())`) or short-circuits
with a typed [`Denial`](https://docs.rs/nest-rs-guards/latest/nest_rs_guards/enum.Denial.html)
the framework maps to a 401 / 403 / 429 response.

`Guard` is a [`Layer`](/fundamentals/) sub-trait shipped by
[`nest-rs-guards`](https://crates.io/crates/nest-rs-guards). One impl,
three transports: declared once, every handler on HTTP, GraphQL, and WS
runs through the same chain.

```rust
#[async_trait]
pub trait Guard: Layer {
    async fn check_http(&self, _req: &mut HttpRequest) -> Result<(), Denial> { Ok(()) }
    async fn check_graphql(&self, _ctx: &GraphqlContext<'_>) -> Result<(), Denial> { Ok(()) }
    async fn check_ws_message(&self, _client: &WsClient, _event: &str, _data: &Value) -> Result<(), Denial> { Ok(()) }
}
```

Override only the `check_*` method(s) where the guard has work to do —
the rest inherit `Ok(())` defaults. A no-op default means "doesn't apply
to this transport", not "skip security".

## A minimal guard

```rust
use nest_rs_guards::prelude::*;
use nest_rs_http::poem::Request as HttpRequest;

#[injectable]
#[derive(Default)]
pub struct RateLimitGuard;

impl Layer for RateLimitGuard {}

#[async_trait]
impl Guard for RateLimitGuard {
    async fn check_http(&self, req: &mut HttpRequest) -> Result<(), Denial> {
        if over_quota(req) {
            return Err(Denial::rate_limited("rate limited", 60));
        }
        Ok(())
    }
}
```

A guard is an `#[injectable]` provider that implements `Guard`. It goes
in some module's `providers = [...]`, and the access graph enforces that
every controller / resolver / gateway referencing it imports the right
module.

## The three scopes

| Scope | Binding | Resolved by |
|-------|---------|-------------|
| **Global** | `App::builder().use_guards_global([guard::<AuthGuard>()])` in `main` | A transport-level interceptor + the per-route shaper |
| **Controller / Resolver / Gateway** | `#[use_guards(AuthGuard, AuthzGuard)]` on the struct (HTTP, WS) or `impl` block (GraphQL) | The container, at mount |
| **Per-handler** | `#[use_guards(AdminGuard)]` beside a verb / `#[query]` / `#[subscribe_message]` | The container, at mount |

Multiple guards listed in one attribute run in declaration order,
outermost first — the first listed sees the request before the second.

```rust
#[controller(path = "/users")]
#[use_guards(AuthGuard, AuthzGuard)]
pub struct UsersController {
    #[inject]
    svc: Arc<UsersService>,
}

#[routes]
impl UsersController {
    #[get("/:id")]
    async fn get(&self, user: Bind<UsersService, Read>) -> Json<User> {
        Json(User::from(&*user))
    }

    #[delete("/:id")]
    #[use_guards(AdminGuard)]
    async fn delete(&self, id: Path<Uuid>) -> Result<()> {
        self.svc.delete(*id).await
    }
}
```

`AdminGuard` runs *on top of* the controller's two guards. Per-handler
is additive, not substitutive.

## TypeId dedup — redeclaration is free

The Layer System dedups by [`TypeId`](https://doc.rust-lang.org/std/any/struct.TypeId.html)
across the three scopes. When the same guard is declared at several
levels, the **broadest scope wins** and the other declarations log a
`warn` at mount but never re-execute.

```rust
// In main.rs
App::builder()
    .use_guards_global([guard::<AuthGuard>(), guard::<AuthzGuard>()])
    .module::<AppModule>()

// In crates/features/src/users/http/controller.rs
#[controller(path = "/users")]
#[use_guards(AuthGuard, AuthzGuard)]   // <- redeclared
pub struct UsersController { /* … */ }
```

`AuthGuard` and `AuthzGuard` run **exactly once** per request, not twice.
The transport-level `GlobalGuardsHttpInterceptor` executes the globals
once at the HTTP boundary, and the per-route `LayersRouteInterceptor`
filters Global-source entries out of the per-route chain to avoid a
second execution.

## Why redeclare on the controller, then?

Defense in depth across crates. A `crates/features` controller is
designed to be portable — the same `UsersController` may be mounted by
`platform-api` today and by another app tomorrow. If the controller
relies only on `use_guards_global([AuthGuard, AuthzGuard])` to be
secure, **forgetting that line in the second app silently turns every
route public**. The compiler will not complain; the controller has no
visible relationship to authentication.

Redeclaring on the controller binds the security policy to the code that
needs it, not to the app that happens to mount it. The cost is one line
and zero runtime overhead (the dedup logic above), the payoff is a
controller that can't accidentally ship unguarded.

The same logic applies symmetrically:

- A `#[public]` handler stays open by intent — the absence of guards is
  the explicit decision, marked by the attribute.
- A handler that needs an extra check beyond the controller's stays
  additive: `#[use_guards(AdminGuard)]` on the method runs on top of the
  controller's `AuthGuard + AuthzGuard`.

<Aside type="caution">
**Dedup wiring varies by transport.**

- **HTTP routes** dedup at the per-route shaper *and* at mount time.
  Controller and method redeclarations of a global guard are skipped
  before the route is ever assembled.
- **WS gateway upgrade** (`#[gateway] + #[use_guards(...)]`) dedups
  against the global chain at mount time — the gateway-scope wrap is
  skipped when the TypeId is already seeded as Global, so the
  transport-level `GlobalGuardsHttpInterceptor` carries the single
  execution.
- **GraphQL resolver / operation** dedups inside
  `run_layered_graphql_chain` — operation guards and resolver-scope
  markers compose through the same `compose_chain` as HTTP, so
  redeclarations across the global / resolver / per-operation scopes
  collapse to one run.

Still queued: per-resolver and per-message dispatch for the
`Interceptor::wrap_graphql` / `wrap_ws`,
`Filter::filter_graphql` / `filter_ws`, and
`ExceptionFilter::catch_graphql` / `catch_ws` methods. The traits
expose those entry points today (one impl, three transports), but only
the HTTP scopes are wired into the per-operation chain — overrides for
GraphQL and WS still ride alongside the global declaration only.
</Aside>

## Attaching request context

A guard often does more than gate access — it produces a value the
handler needs. `AuthGuard` is the canonical case:

```rust
#[async_trait]
impl Guard for AuthGuard<MyStrategy> {
    async fn check_http(&self, req: &mut HttpRequest) -> Result<(), Denial> {
        let claims = self.strategy.authenticate(req).await?;
        req.extensions_mut().insert(claims);   // attach
        Ok(())
    }
}
```

The handler reads it back with `Ctx<T>`:

```rust
use nest_rs_http::Ctx;

#[get("/me")]
async fn me(&self, auth: Ctx<Claims>) -> Json<User> {
    Json(self.svc.find(auth.sub).await?)
}
```

`Ctx<T>` rejects with a 500 if the value is absent — a missing context
means the guard that should have set it never ran. The error names the
missing type so the wiring fix is obvious.

## Reading per-handler metadata

A guard sometimes needs to vary its decision per route. `#[meta(...)]`
attaches a typed payload to the route; `Reflector` reads it back inside
the guard:

```rust
#[derive(Clone)]
pub struct Roles(pub &'static [&'static str]);

#[injectable]
#[derive(Default)]
pub struct RolesGuard;

impl Layer for RolesGuard {}

#[async_trait]
impl Guard for RolesGuard {
    async fn check_http(&self, req: &mut HttpRequest) -> Result<(), Denial> {
        let claims = req.extensions().get::<Claims>().ok_or_else(Denial::forbidden_empty)?;
        let required = Reflector::new(req).get::<Roles>().map(|r| r.0).unwrap_or(&[]);
        if required.iter().any(|r| claims.roles.contains(r)) {
            Ok(())
        } else {
            Err(Denial::forbidden("insufficient roles"))
        }
    }
}
```

```rust
#[get("/admin")]
#[use_guards(AuthGuard, RolesGuard)]
#[meta(Roles(&["admin", "auditor"]))]
async fn admin(&self) -> &'static str { "ok" }
```

<Aside type="caution">
`#[meta(...)]` only reaches a guard bound at controller or per-handler
scope. A global guard runs before routing resolves a handler, so the
reflector finds nothing.
</Aside>

## `#[public]` is metadata, not a framework skip

`#[public]` attaches a [`Public`](https://docs.rs/nest-rs-core/latest/nest_rs_core/struct.Public.html)
marker to the route via the same channel as `#[meta(...)]`. The
framework does **not** act on it — each guard decides what "public"
means for it:

- `AuthGuard` typically still runs and authenticates if a token is
  present (so a downstream policy sees the principal), but does not
  reject when the request is anonymous.
- `AbilityGuard` may still run on a public route to apply visitor rules.

Reading the marker:

```rust
if Reflector::new(req).is_public() { /* relaxed policy */ }
```

This keeps the *what* (run on every route) separate from the *how* (the
guard's own policy).

## Ordering inside a route

Per-route, inner → outer:

```
shaper → interceptors → guards → filters → meta
```

- **Shaper** wraps the handler innermost (e.g. response masking emitted
  by `#[routes]` when a handler takes an `Authorize` parameter).
- **Interceptors** wrap the shaper and can install ambient state across
  the guards (`DbContext` for example installs the SeaORM executor
  before `AbilityGuard` reads from it).
- **Guards** run after interceptors, in declaration order. Inside the
  chain, [`Layer::priority`] is an optional tiebreaker — most guards
  leave it at 0 and rely on declaration order.
- **Filters** map errors coming out of the chain to responses.
- **Meta** is read by guards via `Reflector`.

## GraphQL and WebSocket

`Guard` spans all three transports through `check_http`,
`check_graphql`, and `check_ws_message`. For most apps the global HTTP
declaration is enough — the GraphQL `POST /graphql` and the WS upgrade
are both HTTP requests that pass through the global chain.

GraphQL also exposes `ResolverGuard` (a per-resolver marker) and the
async-graphql `OperationGuard`, both wired through the
`AuthzGraphqlModule` for projects that want a finer per-operation check.
See [Security](/security/) for the bridge between HTTP, GraphQL, and WS
authz.
