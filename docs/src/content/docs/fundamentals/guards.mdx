This page continues the fictional **`blog`** app from
[Modules](/fundamentals/modules/) and
[Providers](/fundamentals/providers/). Once `PostHttpModule` mounts
`PostController`, guards decide *who* reaches each handler ‚Äî and often
attach the principal the handler reads back via `Ctx<T>`.

Four steps on this page:

1. **What a guard is** ‚Äî one trait, three transports, `Ok(())` or
   [`Denial`](https://docs.rs/nest-rs-guards/latest/nest_rs_guards/enum.Denial.html).
2. **Secure the post API** ‚Äî `#[use_guards(AuthGuard, AuthzGuard)]` on
   `PostController`, with the modules that provide those guards.
3. **Three scopes** ‚Äî global, controller, per-handler; additive, not
   substitutive.
4. **Declare on the provider** ‚Äî dedup by `TypeId`; redeclare at the
   controller so security travels with the feature.

## What a guard does

A guard runs *before* the handler. It sees the request first and either
lets it through (`Ok(())`) or short-circuits with a typed `Denial` the
framework maps to 401 / 403 / 429. One trait, three transports ‚Äî the same
`#[injectable]` impl is reachable from HTTP, GraphQL, and WS through the
[Layer System](/fundamentals/).

`Guard` ships in
[`nest-rs-guards`](https://crates.io/crates/nest-rs-guards). It extends
[`Layer`](https://docs.rs/nest-rs-core/latest/nest_rs_core/trait.Layer.html)
so guards plug into dedup-by-`TypeId` and declaration-order chaining.

### A minimal custom guard

```rust title="crates/features/src/post/guard.rs"
use nest_rs_guards::prelude::*;
use nest_rs_http::poem::Request as HttpRequest;

#[injectable]
#[derive(Default)]
pub struct RateLimitGuard;

impl Layer for RateLimitGuard {}

#[async_trait]
impl Guard for RateLimitGuard {
    async fn check_http(&self, req: &mut HttpRequest) -> Result<(), Denial> {
        if over_quota(req) {
            return Err(Denial::rate_limited("rate limited", 60));
        }
        Ok(())
    }
}
```

A guard is an `#[injectable]` provider listed in some module's
`providers = [...]`. The access graph (see
[Providers](/fundamentals/providers/#the-access-graph--wiring-checked-at-boot))
ensures every controller / resolver / gateway that references it imports
a module that builds it.

The trait carries one method per transport. Unimplemented transports
inherit `Ok(())` ‚Äî that means "doesn't apply here", not "skip security".
Other guards on the same route still run.

| Method | Transport | Request shape |
|--------|-----------|---------------|
| `check_http` | HTTP | `&mut HttpRequest` ‚Äî gate and mutate (attach context in extensions) |
| `check_graphql` | GraphQL | `&GraphqlContext` ‚Äî read-only; seed context upstream |
| `check_ws_message` | WS | per inbound message, after the upgrade |

```rust
#[async_trait]
pub trait Guard: Layer {
    async fn check_http(&self, _req: &mut HttpRequest) -> Result<(), Denial> { Ok(()) }
    async fn check_graphql(&self, _ctx: &GraphqlContext<'_>) -> Result<(), Denial> { Ok(()) }
    async fn check_ws_message(&self, _client: &WsClient, _event: &str, _data: &Value)
        -> Result<(), Denial> { Ok(()) }
}
```

## Secure the post API

After [Modules](/fundamentals/modules/#extract-a-feature), `PostController`
lives under `post/http/`. Binding guards on the struct is how you gate
every route on that controller ‚Äî and declare the auth modules the access
graph must reach.

```rust title="crates/features/src/post/http/controller.rs"
use std::sync::Arc;
use nest_rs_authz::Read;
use nest_rs_http::{controller, routes};
use nest_rs_seaorm::Bind;
use poem::web::{Json, Path};
use poem::Result;
use uuid::Uuid;

use crate::authn::AuthGuard;
use crate::authz::AuthzGuard;
use crate::post::{AdminGuard, Post, PostService};

#[controller(path = "/posts")]
#[use_guards(AuthGuard, AuthzGuard)]
pub struct PostController {
    #[inject]
    svc: Arc<PostService>,
}

#[routes]
impl PostController {
    #[get("/:id")]
    async fn get(&self, post: Bind<PostService, Read>) -> Json<Post> {
        Json(Post::from(&*post))
    }

    #[delete("/:id")]
    #[use_guards(AdminGuard)]
    async fn delete(&self, id: Path<Uuid>) -> Result<()> {
        self.svc.delete(*id).await
    }
}
```

`AdminGuard` runs *on top of* the controller's two guards. Per-handler
binding is **additive** ‚Äî it does not replace the controller list. A route
with no guards stays open *by intent*; omitting `#[use_guards(...)]` is
the explicit decision.

The HTTP adapter imports the auth bridges transitively ‚Äî same pattern as
`UsersHttpModule` in the repo:

```rust title="crates/features/src/post/http/module.rs"
#[module(
    imports = [PostModule, AuthzHttpModule],
    providers = [PostController],
)]
pub struct PostHttpModule;
```

```rust title="crates/features/src/authn/module.rs"
#[module(
    imports = [nest_rs_authn::AuthnModule::for_root(None)],
    providers = [AppJwtStrategy, AuthGuard],
)]
pub struct AuthnModule;
```

```rust title="crates/features/src/authz/http/module.rs"
#[module(
    imports = [AuthzModule],
    providers = [AuthzGuard],
)]
pub struct AuthzHttpModule;
```

`AdminGuard` on top. `AuthGuard` attaches `Claims`; `AuthzGuard` builds
the ambient `Ability` handlers and `Bind` read through.

## The three scopes

A guard binds at one of three scopes. The container resolves all three
from the same provider ‚Äî declare once, choose *where* it runs.

| Scope | Binding | Resolved by |
|-------|---------|-------------|
| **Global** | `App::builder().use_guards_global([guard::<AuthGuard>()])` in `main` | Transport-level interceptor + per-route shaper |
| **Controller / Resolver / Gateway** | `#[use_guards(AuthGuard, AuthzGuard)]` on the struct | Container, at mount |
| **Per-handler** | `#[use_guards(AdminGuard)]` beside a verb / `#[query]` / `#[subscribe_message]` | Container, at mount |

Multiple guards in one attribute run in **declaration order**, outermost
first ‚Äî the first listed sees the request before the second.

```rust title="apps/blog/src/main.rs"
use nest_rs_guards::{AppBuilderGuardsExt, guard};
use crate::authn::AuthGuard;
use crate::authz::AuthzGuard;

App::builder()
    .use_guards_global([guard::<AuthGuard>(), guard::<AuthzGuard>()])
    .module::<BlogModule>()
```

Across scopes the chain composes **global ‚Üí controller ‚Üí handler**,
outermost first. `Layer::priority` is an optional tiebreaker when
declaration order cannot express intent ‚Äî most guards leave it at `0`.

```mermaid
flowchart TB
  G[global guards] --> C[controller guards]
  C --> H[handler guards]
  H --> R[handler]
```

## Ordering inside a route

Per-route, inner ‚Üí outer:

```
shaper ‚Üí interceptors ‚Üí guards ‚Üí filters ‚Üí meta
```

- **Interceptors** wrap the shaper and install ambient state *before*
  guards run (`DbContext` sets the SeaORM executor so `AuthzGuard` can
  read from it).
- **Guards** run after interceptors, in declaration order.
- **Meta** is read by guards via `Reflector`.

<Aside>
**Global** interceptors wrap *outside* global guards ‚Äî the data context
must install the executor around the guards too. Controller- and
handler-bound interceptors wrap *inside* the controller's guards.
</Aside>

```mermaid
flowchart LR
  S[shaper] --> I[interceptors]
  I --> G[guards]
  G --> F[filters]
  F --> M[meta]
  M --> H[handler]
```

## Attach context and route metadata

Guards often do more than gate ‚Äî they produce values the handler needs.
Attach on the way in; read with `Ctx<T>`:

```rust
#[async_trait]
impl Guard for AuthGuard<MyStrategy> {
    async fn check_http(&self, req: &mut HttpRequest) -> Result<(), Denial> {
        let claims = self.strategy.authenticate(req).await?;
        req.extensions_mut().insert(claims);
        Ok(())
    }
}

#[get("/me")]
async fn me(&self, auth: Ctx<Claims>) -> Json<Post> {
    Json(self.svc.find_by_author(auth.sub).await?)
}
```

`Ctx<T>` rejects with 500 if the value is absent ‚Äî a missing context
means the guard that should have set it never ran. Store an `Arc<_>` if
the value is large.

`AuthzGuard` reads `Claims` left by `AuthGuard` to build the ambient
`Ability`. That is why order matters:
`#[use_guards(AuthGuard, AuthzGuard)]`, not the reverse.

For per-route policy ‚Äî required roles, rate-limit quotas ‚Äî `#[meta(...)]`
attaches a typed payload; `Reflector` reads it inside the guard:

```rust
#[get("/admin/audit")]
#[use_guards(AuthGuard, RolesGuard)]
#[meta(Roles(&["admin", "auditor"]))]
async fn audit(&self) -> &'static str { "ok" }
```

`#[meta(...)]` only reaches guards at controller or per-handler scope. A
global guard runs before routing resolves a handler, so the reflector
finds nothing.

`#[public]` uses the same channel. The framework does **not** act on it ‚Äî
each guard decides what "public" means (`AuthGuard` may authenticate when
a token is present but not reject anonymous callers; `AuthzGuard` may
still apply visitor rules). Marking a route public does not strip its
guards; it tells them to relax.

## Declare on the provider

The Layer System dedups every layer by
[`TypeId`](https://doc.rust-lang.org/std/any/struct.TypeId.html) across
the three scopes. When the same guard is declared at several levels, the
**broadest scope wins**; narrower declarations log a `warn` at mount but
never re-execute.

```rust title="apps/blog/src/main.rs"
App::builder()
    .use_guards_global([guard::<AuthGuard>(), guard::<AuthzGuard>()])
    .module::<BlogModule>()
```

```rust title="crates/features/src/post/http/controller.rs"
#[controller(path = "/posts")]
#[use_guards(AuthGuard, AuthzGuard)]
pub struct PostController { /* ... */ }
```

`AuthGuard` and `AuthzGuard` run **exactly once** per request, not twice.
The controller's two extra lines cost zero runtime overhead and one `warn`
at boot.

**Declare layers on the provider that needs them, not only at the app
boundary.** A `crates/features` controller is designed to be portable ‚Äî if
it relies only on `use_guards_global([...])` to be secure, forgetting
that line in a second app silently turns every route public. The compiler
will not complain. Redeclaring on `PostController` binds the policy to the
code that needs it. The same rule applies to interceptors and filters.

```mermaid
flowchart TB
  MAIN["main: use_guards_global([AuthGuard, AuthzGuard])"]
  PC["PostController: #[use_guards(AuthGuard, AuthzGuard)]"]
  MAIN -. "dedup ‚Äî runs once" .-> RUN[per-request chain]
  PC -. "warn at boot, zero extra cost" .-> RUN
```

## HTTP, GraphQL, and WebSockets

For most apps the global HTTP declaration is enough: GraphQL
`POST /graphql` and the WS upgrade are both HTTP requests that pass
through the global chain.

Each transport method runs at a different seam:

- `check_http` ‚Äî on `&mut HttpRequest` before routing.
- `check_graphql` ‚Äî inside the resolver's per-operation chain (context
  is read-only).
- `check_ws_message` ‚Äî once per inbound message, after the connection
  upgrade.

GraphQL and WS need a bridge one level above the resolver / gateway
because the per-operation context does not carry the poem request:

- **GraphQL** ‚Äî a `GraphqlOperationGuard` (e.g. `GraphqlAbilityBridge`)
  re-runs the HTTP guard chain on `POST /graphql` and seeds `Ability`
  for the operation. Per-resolver gating uses a marker like
  `GraphqlAuthGuard`.
- **WS** ‚Äî a `SocketContext` (e.g. `WsDataContext`) captures pool +
  `Ability` on upgrade and re-installs them around each message.
  Per-message gating goes through `check_ws_message`.

On a resolver or gateway, `#[use_guards(...)]` often carries a **marker
guard** ‚Äî a struct whose job is to declare a dependency on the seeded
context so the access graph can validate it. Omitting the matching authz
module fails boot with a clear error naming the missing guard.

```rust
#[resolver]
#[use_guards(GraphqlAuthGuard)]
impl PostResolver { /* ... */ }
```

```rust
#[gateway(path = "/posts/live")]
#[use_guards(AuthGuard, AuthzGuard)]
pub struct PostGateway { /* ... */ }

#[messages]
impl PostGateway {
    #[subscribe_message("comment")]
    #[use_guards(WsAuthGuard)]
    async fn comment(&self, msg: CommentInput) -> Reply { /* ... */ }
}
```

See [Security / per-transport bridges](/security/authorization/per-transport-bridges/)
for the full HTTP / GraphQL / WS wiring, and
[WebSockets / Guards](/websockets/guards/) for the two WS scopes
(connection vs message).

<Aside type="note" title="What to remember">
- **Guard** ‚Äî `#[injectable]` provider; `Ok(())` passes, `Denial`
  short-circuits with a typed status.
- **Three scopes** ‚Äî global, controller, handler; compose outermost-first;
  per-handler is additive.
- **Declare on the provider** ‚Äî `#[use_guards(...)]` on `PostController`
  binds security to the feature; global + controller dedup to one run.
- **Order matters** ‚Äî `AuthGuard` before `AuthzGuard`; interceptors
  before guards so ambient state is installed first.
</Aside>
