Five layer families wrap every request: **guards** (gate access), **pipes**
(edge conversion), **interceptors** (cross-cutting, e.g. transactions),
**filters** and **exception-filters** (error mapping). This page is the one
place that explains how they compose ŌĆö every other page links here instead of
re-explaining it.

## One pool per family

Declaring a layer at any scope ŌĆö **global** (`use_*_global`), **controller**
(on the struct), or **handler** (beside the verb) ŌĆö contributes to **one pool
per family**, deduplicated by `TypeId` through a single `compose_chain`. The
layer executes **exactly once per request**. Broadest scope wins;
`#[force_guards]` is the opt-in to re-run.

Scope never multiplies executions ŌĆö it chooses the **execution site**. The
teachable rule:

> Global = around the whole HTTP process. Scoped = around your handler. Either
> way, once.

You can watch the dedup happen. Declare `AuthnGuard` both globally
(`use_guards_global`) and on a controller (`#[use_guards(AuthnGuard, ŌĆ”)]`), and
boot logs the redundant declaration it dropped ŌĆö the guard still runs exactly
once:

```text
DEBUG nest_rs::layers: redundant layer declaration deduped
  layer="AuthnGuard<JwtStrategy<Claims>>" kept="global" skipped="controller"
  hint="broadest scope wins; use `#[force_*]` to re-run"
```

## Execution sites

Each family runs at the site that matches its nature. The pool is the same; the
site differs:

| Family | Global scope | Controller / handler scope |
|---|---|---|
| Guard | `RouteShaper` (post-routing ŌĆö reads `#[public]`) | same site |
| Pipe | `RouteShaper` | `RouteShaper` |
| ExceptionFilter | route site (closest to the handler) | route site |
| Interceptor | transport edge (before auth ŌĆö sees 404s, denials) | around the handler, inside guards |
| Filter | transport edge | around the handler, inside guards |

Global guards are **pooled** globally but **execute** post-routing at the
`RouteShaper`, because a guard reads `#[public]` route data that only exists
after routing ŌĆö they do **not** run "before routing".

Two ways to be transport-wide, deliberately distinct: `use_*_global` adds to
the **pool** (TypeId-deduped against narrower scopes); `#[interceptor]` is
**infrastructure** a module import auto-mounts at a fixed band (off the pool ŌĆö
`DbContext`, tracing, timing).

## Nesting order

Per route, innermost ŌåÆ outermost: **handler ŌåÆ ability shaper ŌåÆ exception-filter
pool ŌåÆ scoped filters ŌåÆ scoped interceptors ŌåÆ `RouteShaper` (guard pool ŌåÆ pipe
pool) ŌåÆ `#[meta]`/`#[public]` route data**. At the transport edge
(innermost ŌåÆ outermost band): **routing ŌåÆ DbContext (ŌłÆ10) ŌåÆ global filter pool
(50) ŌåÆ global interceptor pool (90) ŌåÆ infra `#[interceptor]` (100)**.

Interceptors sit outside filters; exception-filters sit closest to the handler
ŌĆö at both sites. `Layer::priority` orders entries *within* a site, never across
sites.

## `#[public]` and `#[meta]`

`#[public]` and `#[meta(EXPR)]` are not layers ŌĆö they are **route metadata** a
guard reads after routing. `#[public]` marks a route as anonymous-admissible
(the guard admits it through); `#[meta]` attaches per-handler data a guard reads
via `nest_rs_http::Reflector`. A denial is an `Ok(4xx)` response, never an
`Err` ŌĆö filters never see it; global interceptors do.

## What fails at boot

Wiring mistakes fail the boot with a named error, never a silent drop.

If a provider injects a layer whose module the app never imported, the
**access graph** fails the boot naming the exact fix:

```text
module access violation: `PostsController` (in module `PostsHttpModule`)
depends on `AuthzGuard`, but `PostsHttpModule` imports no module that
provides it. `AuthzGuard` is provided by `AuthzHttpModule` ŌĆö add
`AuthzHttpModule` to `#[module(imports = [...])]` of `PostsHttpModule`, or
route the dependency through a module `PostsHttpModule` already imports.
```

A global layer spec that cannot resolve (its provider's module is not imported)
fails the boot through `HttpBootCheck`, naming the type. An imperative
self-mount under active global guards fails too, unless
`fail_secure_strict = false` downgrades it to a warn (it defaults to `true`).

<Aside type="caution">
These are boot-time failures, by design. A missing guard module is a compile-
then-boot break, not a request that silently runs unguarded. Fail-secure is the
point: forgetting to wire authz cannot ship.
</Aside>
