## In a dev container (recommended)

The repository ships a dev container that provisions Rust, Postgres, Redis and
the dev tooling in one step. It is the fastest path to a working setup.

<Steps>

1. Install [Docker](https://docs.docker.com/get-docker/) and the VS Code
   [Dev Containers](https://marketplace.visualstudio.com/items?itemName=ms-vscode-remote.remote-containers)
   extension.

2. Clone the repository.

   ```bash
   git clone https://github.com/NestRS/NestRS.git nestrs
   cd nestrs
   ```

3. Open the folder in VS Code and accept **Reopen in Container** when prompted.

4. Run the baseline app.

   ```bash
   just dev
   # → http://localhost:3000 returns "Hello World"
   ```

</Steps>

`just dev` runs under `bacon` — every save triggers an incremental rebuild and
a restart. Edit a handler, save, hit the endpoint again.

## On your own machine

If you prefer a local toolchain:

```bash
# Rust 1.75 or newer
curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh

# Dev tooling
cargo install --locked just bacon cargo-nextest
rustup component add llvm-tools-preview

# Clone and run
git clone https://github.com/NestRS/NestRS.git nestrs
cd nestrs
just dev
```

Apps that need Postgres (`platform-api`, `platform-auth`) or Redis
(`platform-worker`) read their connection string from
`NESTRS_DATABASE__URL` / `NESTRS_QUEUE__URL`. The bare `hello`, `mcp`
and `chat` examples need neither.

## Run the examples

The workspace ships several example apps. Each is a separate runnable binary;
they share the same workspace and the same building blocks. Run one at a time —
each serves on `http://localhost:3000`.

```bash
just dev                  # bare hello baseline

just dev platform-auth    # OAuth2 / JWT issuer        (requires `just db up` once)
just dev platform-api     # REST + GraphQL + OpenAPI   (requires `just db up` once)
just dev platform-worker  # background jobs (headless)

just dev mcp              # MCP server

just dev chat             # WebSocket gateway
```

## Your first endpoint

A complete service — a provider, a controller, and the root module that
lists them directly:

<FileTree>
- src/
  - main.rs
  - module.rs
  - service.rs
  - controller.rs
</FileTree>

```rust title="src/service.rs"
use nest_rs_core::injectable;

#[injectable]
#[derive(Default)]
pub struct HelloService;

impl HelloService {
    pub fn greeting(&self) -> String { "Hello World".to_string() }
}
```

```rust title="src/controller.rs"
use std::sync::Arc;
use nest_rs_http::{controller, routes};
use crate::service::HelloService;

#[controller(path = "/")]
pub struct HelloController {
    #[inject]
    svc: Arc<HelloService>,
}

#[routes]
impl HelloController {
    #[get("/")]
    async fn hello(&self) -> String {
        self.svc.greeting()
    }
}
```

```rust title="src/module.rs"
use nest_rs_core::module;
use nest_rs_http::HttpModule;
use crate::{controller::HelloController, service::HelloService};

#[module(
    imports = [HttpModule::for_root(None)],
    providers = [HelloService, HelloController],
)]
pub struct HelloModule;
```

`HttpModule::for_root(None)` listens on host `0.0.0.0` and port `3000` by
default — override with `NESTRS_HTTP__HOST` / `NESTRS_HTTP__PORT`, or pin a
different port in code with
`HttpModule::for_root(HttpConfig { port: 3002, ..Default::default() })`.

```rust title="src/main.rs"
use nest_rs_core::App;
mod controller;
mod module;
mod service;
use module::HelloModule;

#[tokio::main]
async fn main() -> anyhow::Result<()> {
    App::builder()
        .module::<HelloModule>()
        .build()
        .await?
        .run()
        .await
}
```

The HTTP transport is activated by importing `HttpModule::for_root(None)`
in `HelloModule`; `main` carries no transport at all and is identical
across every app in the workspace.

```toml title="Cargo.toml"
[dependencies]
anyhow         = "1"
tokio          = { version = "1", features = ["macros", "rt-multi-thread"] }
nest-rs-core   = "0"
nest-rs-http   = "0"
```

(`nest-rs-*` is the kebab-case name on crates.io. In code you still write
`use nest_rs_core::App;` — the lib names keep the `nestrs_*` form.)

```bash
cargo run
# → GET http://localhost:3000 returns "Hello World"
```

## One import instead of many — the `nestrs` umbrella

The example above pulls each surface from its own crate (`nest-rs-core`,
`nest-rs-http`, …) — that split is the public, versioned source of truth
and keeps working. For app code that just wants the everyday decorators
without naming each crate, the optional `nestrs` umbrella crate collects
them behind Cargo features:

```toml title="Cargo.toml"
[dependencies]
nest-rs = { version = "0", features = ["http", "database", "seaorm"] }
```

```rust title="src/controller.rs"
use std::sync::Arc;
use nest_rs::prelude::*;
use crate::service::HelloService;

#[controller(path = "/")]
pub struct HelloController {
    #[inject]
    svc: Arc<HelloService>,
}

#[routes]
impl HelloController {
    #[get("/")]
    async fn hello(&self) -> String {
        self.svc.greeting()
    }
}
```

`use nest_rs::prelude::*;` brings in `App`, `AppBuilder`, `Container`,
`Module`, `hooks`, `injectable`, `module`, and — when the `http` feature
is on — `controller`, `routes`, `http_code`, `redirect`,
`response_header`, `input`, `Valid`, `Scoped`, `Ctx`, `ClientIp`,
`RawBody`, `Reflector`, `HttpConfig`, `HttpModule`, plus the poem
extractors `Json`, `Path`, `Query`. Verb attributes (`#[get]`, `#[post]`,
…) come in via `#[routes]`.

| Feature flag | Surface it pulls in |
|--------------|---------------------|
| `http` (default) | `nest-rs-http` — controllers, routes, extractors |
| `config` (default) | `nest-rs-config` — typed config + `.env` cascade |
| `database` / `seaorm` | the data layer (`nest-rs-database` seam, SeaORM impl) |
| `graphql`, `ws`, `mcp` | the corresponding transport |
| `queue` / `redis` | the queue seam + the Redis backend |
| `schedule`, `events` | cron jobs, in-process pub/sub |
| `authn`, `authz` | the auth machinery |
| `openapi`, `health`, `throttler`, `server-timing`, `opentelemetry` | the matching module |
| `testing` | `nest-rs-testing` — `TestApp`, `EphemeralDatabase` |

Pick the features the app uses; nothing else compiles. The per-crate
imports keep working unchanged — the umbrella is opt-in, not a
replacement.

## What just happened

- **`#[injectable]`** marked `HelloService` as a provider; `App::build()`
  constructs it once and shares it as `Arc<HelloService>`.
- **`#[controller]` + `#[routes]`** generated a poem `Endpoint` that mounts
  the verb methods on the controller's path.
- **`#[module]`** registered both with the DI container; `imports = [...]` on
  `HelloModule` opened access from the root.
- **`App::new`** verified the access graph — every `#[inject]` resolves to
  something the importing module can reach — and returned `Result`: a bad
  wiring fails here with a clear error.
