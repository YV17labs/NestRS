## Why developers pick NestRS

<CardGrid>
  <Card title="Native throughput" icon="rocket">
    **~463k req/s** on 2 cores, **~213k per core**, sub-millisecond p99,
    no GC pauses. The same `hyper`/`tokio` core as the fastest Rust web
    frameworks — with a structure on top.
  </Card>
  <Card title="An order of magnitude less RAM" icon="laptop">
    **~4 MB idle, ~6 MB under load** — versus ~80–120 MB for an
    equivalent Node service. Smaller instances, higher density, a
    materially lighter cloud bill.
  </Card>
  <Card title="Boots in milliseconds" icon="bars">
    Each deployable is a static native binary with no runtime to warm up —
    autoscaling and cold starts stop hurting.
  </Card>
  <Card title="Types you don't fight" icon="approve-check">
    Rust types end to end — entity, DTO, route handler, dataloader, GraphQL
    schema, OpenAPI doc. No `any`, no `unknown`, no `as` casts at the
    boundaries, no runtime "undefined is not a function."
  </Card>
  <Card title="Declarative, decorator-driven" icon="puzzle">
    `#[module]`, `#[controller]`, `#[resolver]`, `#[processor]`,
    `#[gateway]`, `#[cron_job]`, `#[mcp]`. Wiring is declarative — you write
    logic, not boilerplate.
  </Card>
  <Card title="Verified before it serves" icon="warning">
    The DI graph is checked at boot — a bad import fails startup with a
    clear error naming the missing wire. No reflection, no runtime
    "cannot resolve dependency" five minutes after deploy.
  </Card>
  <Card title="Secure & transactional by composition" icon="approve-check-circle">
    Authn, authorization, row-level filtering, response masking, transaction
    scope — activated by importing a module, not by remembering to call
    them. Forgetting to redact a field becomes structurally hard.
  </Card>
  <Card title="Batteries included, opt-in by crate" icon="setting">
    HTTP, GraphQL, OpenAPI, WebSockets, Redis-backed queues, scheduling,
    an event bus, MCP, CASL-style authorization, health probes,
    OpenTelemetry — each an opt-in crate, so a worker compiles no HTTP
    stack at all.
  </Card>
</CardGrid>

## How fast, exactly

The same "Hello World" service — a provider, a controller, a module — built
once in NestRS and once in NestJS 11 on Node.js 20, under identical `wrk` load
(2 cores, `-t2 -c64 -d20s`).

## What a module looks like

A minimal slice — a provider, a controller, and a `#[module]` that lists them.
The DI graph is built and verified at boot; each app under `apps/` compiles to
its own native binary. For a full vertical slice (core port plus HTTP/GraphQL adapters), start with the
[tutorial](/tutorial/).

```rust title="apps/app/src/hello/service.rs"
use nestrs_core::injectable;

#[injectable]
#[derive(Default)]
pub struct HelloService;

impl HelloService {
    pub fn greeting(&self) -> &'static str { "Hello World" }
}
```

```rust title="apps/app/src/hello/controller.rs"
use std::sync::Arc;
use nestrs_http::{controller, routes};
use super::service::HelloService;

#[controller(path = "/")]
pub struct HelloController {
    #[inject]
    svc: Arc<HelloService>,
}

#[routes]
impl HelloController {
    #[get("/")]
    async fn hello(&self) -> &'static str {
        self.svc.greeting()
    }
}
```

```rust title="apps/app/src/hello/module.rs"
use nestrs_core::module;
use super::{controller::HelloController, service::HelloService};

#[module(providers = [HelloService, HelloController])]
pub struct HelloModule;
```

## One domain, several focused binaries

NestRS is built for **microservice-shaped backends without the old ceremony** —
several deployables, **one shared feature library**, and no obligation to split
into many repositories while the team is still small.

**Entities, services, and policy are written once.** Each runnable has a **clear
responsibility** — some sit on a **public edge** (surfaces clients and partners
call), others stay **non-public** (background work, off the request path). Processes
stay lean; your own services stay **loosely coupled** (shared contracts and data,
not chatty RPC, no platform team or mesh required to get started).

## Scale each role on its own signal

Splitting by responsibility is only the first step. In production you **scale
horizontally per role** — the same binary, more replicas when the pressure moves:
API capacity follows **demand**, worker capacity follows **queue depth**. There is
no single “scale the monolith” knob; each deployable gets its own replica count.

**Alpha.** The API still shifts, rough edges remain — NestRS is not
production-ready yet. Stars and early feedback on
[Discussions](https://github.com/NestRS/NestRS/discussions) shape what
follows.
