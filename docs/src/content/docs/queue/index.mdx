Background work shares the rest of the framework's shape. A queue
handler is a method on any `#[injectable]` provider, tagged
`#[process(queue = "...", concurrency, retries)]` inside a `#[processor]`
impl block. Importing `QueueWorkerModule` in the worker app activates
the runtime at boot, drains the discovered methods, and spawns one
worker per method. A producer-only app skips that import and only gets
the connection — same crate, same types, one less module.

The abstractions — `Job`, `Processor`, `ProcessMethod`, the `#[processor]`
macro — ship in **`nest-rs-queue`**. The first-class storage, **Redis**,
ships in **`nest-rs-redis`** (built on
[`apalis`](https://docs.rs/apalis)): the `QueueConnection` producer,
the `QueueWorker` transport, and the `QueueModule` / `QueueWorkerModule`
activation seams. An app depends on both: `nest-rs-queue` for the macro
and abstractions, `nest-rs-redis` for the Redis-bound types.

## A first queue handler

A regular `#[injectable]` service with one decorated method. The
`#[processor]` attribute marks the impl block; `#[process]` marks the
method as a queue consumer.

```rust title="crates/features/src/audio/queue/processor.rs"
use std::sync::Arc;

use anyhow::Result;
use nest_rs_core::injectable;
use nest_rs_queue::processor;

use crate::audio::{AudioService, TranscodeJob};

#[injectable]
pub struct AudioJobs {
    #[inject]
    svc: Arc<AudioService>,
}

#[processor]
impl AudioJobs {
    #[process(queue = "audio", concurrency = 5, retries = 3)]
    async fn transcode(&self, job: TranscodeJob) -> Result<()> {
        self.svc.transcode(&job.file).await
    }
}
```

The processor is thin: it owns no business logic of its own — it hands
`job.file` to `AudioService::transcode` and propagates the result. The
same `AudioService` is what an HTTP producer calls to enqueue, so a
single audited choke point describes everything the feature can do.
Adapters carry transport mechanics; the service carries intent.

- **`#[injectable]`** makes `AudioJobs` a regular DI provider.
- **`#[processor]`** on the impl block orchestrates the per-method
  `#[process]` attributes and submits one queue consumer per decorated
  method.
- **`#[process(queue, concurrency, retries)]`** binds the method to a
  Redis queue name (stringly-typed, the known cost), bounds the
  in-flight jobs, and gives an apalis retry budget. The job type is
  read from the method's second parameter.

## Several methods on the same provider

The pattern the framework is built for: one `#[injectable]` declares
the deps once, multiple decorated methods share them. Different queues,
different concurrencies, same service:

```rust title="crates/features/src/audio/queue/processor.rs"
#[injectable]
pub struct AudioJobs {
    #[inject] svc_audio: Arc<AudioService>,
    #[inject] svc_media: Arc<MediaService>,
}

#[processor]
impl AudioJobs {
    #[process(queue = "audio", concurrency = 5, retries = 3)]
    async fn transcode(&self, job: TranscodeJob) -> Result<()> {
        self.svc_audio.transcode(&job.file).await
    }

    #[process(queue = "audio.preview", concurrency = 10, retries = 1)]
    async fn preview(&self, job: PreviewJob) -> Result<()> {
        self.svc_media.preview(&job.file).await
    }

    #[process(queue = "audio.cleanup")]
    async fn cleanup(&self, job: CleanupJob) -> Result<()> {
        self.svc_media.cleanup(job.older_than).await
    }
}
```

`#[process(queue = "audio.cleanup")]` omits the optional `concurrency`
(default `1`) and `retries` (default `0`).

Three inventory entries — `AudioJobs::transcode`, `AudioJobs::preview`,
`AudioJobs::cleanup` — all pointing at the same `AudioJobs` instance.
Same `Arc<AudioService>`, same `Arc<MediaService>`, three queues. Each
method still delegates: the processor is a router, never a place where
business logic accumulates.

<Aside>
A single `#[processor]` impl block per struct (multiple would conflict
at the discovery layer). Put every queue handler of a service in the
same block — same convention as `#[routes]` and `#[scheduled]`.
</Aside>

## Wire it in

Two modules, two roles:

- **`QueueModule::for_root(None)`** — provides the shared
  `QueueConnection`. Every app (producer *and* consumer) needs it.
- **`QueueWorkerModule`** — activates the consumer runtime. At boot the
  framework drains the discovered `#[process]` methods, spawns one
  apalis worker per method, and shuts them down with the app. A
  producer-only app does NOT import this.

```rust title="apps/publish-worker/src/module.rs"
use nest_rs_core::module;
use nest_rs_redis::{QueueModule, QueueWorkerModule};

use features::audio::{AudioModule, AudioQueueModule};

#[module(imports = [
    ConfigModule::for_root(),
    QueueModule::for_root(None),
    QueueWorkerModule,
    AudioModule,
    AudioQueueModule,
])]
pub struct PublishWorkerModule;
```

`main` is the universal skeleton:

```rust title="apps/publish-worker/src/main.rs"
use anyhow::Result;
use nest_rs_core::App;

use publish_worker::PublishWorkerModule;

#[tokio::main]
async fn main() -> Result<()> {
    App::builder()
        .module::<PublishWorkerModule>()
        .build()
        .await?
        .run()
        .await
}
```
