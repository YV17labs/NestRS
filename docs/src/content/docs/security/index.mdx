Security in nestrs splits into two concerns wired by two composable
layers. Authentication establishes *who* the caller is. Authorization
declares *what* they may do, then enforces it on every read, every
by-id write, and every response body before it leaves the process.
Bind the two guards on the controller and the framework does the rest
— there is no per-handler checklist.

The authn layer leans on [`jsonwebtoken`](https://docs.rs/jsonwebtoken),
[`argon2`](https://docs.rs/argon2), and the
[`oauth2`](https://docs.rs/oauth2) crate. The authz layer is the
framework's own CASL-style engine, with row-level filtering bridged into
[SeaORM](https://docs.rs/sea-orm).

## Two concerns, two layers

| Concern | Layer | Who answers it |
|---------|-------|----------------|
| Who is calling? | Authentication | A [`Strategy`](/security/authentication/) turns the request into a principal (or returns a 401 / OAuth redirect) |
| What may they do? | Authorization | An [`Ability`](/security/authorization/) declares allowed actions; the framework applies it to queries, by-id loads, and response bodies |

Each layer ships its own guard. Bind them in order — the authn guard
runs first and attaches the principal; the authz guard reads it and
installs the ambient `Ability`.

```rust title="crates/features/src/users/http/controller.rs"
#[controller(path = "/users")]
#[use_guards(AuthGuard, AuthzGuard)]
pub struct UsersController {
    #[inject]
    svc: Arc<UsersService>,
}
```

That's the whole opt-in surface for HTTP. GraphQL and WebSockets get
matching bridges that re-establish the ambient ability at the dispatch
point — see [Per-transport bridges](/security/authorization/per-transport-bridges/).

## What you get once both are mounted

- **Every read is filtered.** The
  [`Repo`](/database/repo-and-executor/) gateway every service uses
  joins the ambient ability's
  [`condition_for`](/security/authorization/row-level-filtering/) into
  the SQL — a member querying `Users::find()` gets
  `WHERE org_id = $caller_org_id` appended.
- **Every by-id write checks access.** `Bind<S, A>` calls
  `Service::access(id)`; missing rows return `404`, denied rows return
  `403` — see [By-id binding](/security/authorization/by-id-binding/).
- **Every response is masked.** The
  [`Authorize` shaper](/security/authorization/response-masking/) runs
  after a 2xx handler: fields the caller cannot read become `null`, and
  `retain_wire_keys` strips back anything outside the wire DTO so an
  unrestricted grant cannot leak a `#[expose(skip)]` column.

The wiring sketch for an app that serves all three transports:

```rust title="apps/publish-api/src/module.rs"
#[module(
    imports = [
        DatabaseModule::for_root(None),
        HttpModule::for_root(None),
        GraphqlModule::for_root(None),
        AuthnModule,
        AuthzHttpModule,
        AuthzGraphqlModule,
        AuthzWsModule,
        UsersHttpModule,
        UsersGraphqlModule,
        UsersWsModule,
    ],
)]
pub struct PublishApiModule;
```

<Aside>
A handler opts **out** by not importing the transport's authz module —
public endpoints do exactly that and lose every layer transitively.
Visitor-rule policy then lives in your `AbilityFactory`, not in
`AuthGuard`.
</Aside>
